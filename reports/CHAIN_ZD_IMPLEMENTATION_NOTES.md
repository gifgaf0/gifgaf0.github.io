### Mapping the addendum's notation to the code

The addendum uses **Lower / Upper / Sub-A / ПҖ_A / y_lower / y_upper /
r_upper / r_lower** terminology. Mapping to the actual sedenion code:

- **Lower / Sub-A** = coordinates `eвӮҖ..eвӮҮ` (the first octonion copy);
  for any sedenion vector `v вҲҲ F_p^16`, `ПҖ_A(v) = (v[0], вҖҰ, v[7])`.
- **Upper / ker ПҖ_A** = coordinates `eвӮҲ..eвӮҒвӮ…` (the second octonion
  copy).
- The product block structure quoted in the addendum
  (LowerГ—LowerвҶ’Lower, LowerГ—UpperвҶ’Upper, UpperГ—LowerвҶ’Upper,
  UpperГ—UpperвҶ’Lower) is the Cayley-Dickson doubling rule for the
  octonion вҶ’ sedenion step. Verified against the table built in
  `sedenion_Fp.py:build_cd_table(4)`.
- **y** = the long-term secret `s` (output of `keygen`,
  `sk = s вҲҲ (F_p^16)^k`).
- **r** = the per-encaps randomness drawn in `encaps`
  (sqt_slwe.py:232 or slwe_toy.py:280), a length-k vector of
  sedenions; `r_lower` and `r_upper` are the two octonion halves of
  each component.
- **x вҲҲ Sub-A** = a probing element with `x[8:16] = 0` whose `xВ·s`
  product the attacker can observe. There is currently **no
  oracle** in the implemented scheme that returns `xВ·s` (or
  `ПҖ_A(xВ·s)`) for an attacker-chosen `x` вҖ” see R1 finding below.

### R1 вҖ” y_lower probeable through a clean Sub-A channel?

**Verdict: no current code path matches the literal R1 attack pattern,
but the principle is load-bearing and should be added to the spec
before any future API extension.**

Reasoning. R1 forbids any code path where the attacker can observe
ПҖ_A(xВ·s) for an attacker-chosen `x вҲҲ Sub-A` with `r_upper` zero,
predictable, or not yet mixed. In the present scheme:

- The attacker never directly multiplies their own `x` by the secret
  `s`. The two product evaluations involving `s` are
  `b = AВ·s + e` (computed in keygen, output via pk) and
  `вҹЁs, c1вҹ©_norm` (computed inside decaps, never output вҖ” it feeds
  the m_dec branch and is discarded).
- `pk = (A, b)` exposes `b = AВ·s + e` where `A` is generated by the
  protocol (Singer orbit, then `singer_a_randomized` in the slwe_toy
  path); `A` is not attacker-chosen for a fixed pk. So the
  attacker's R1 lever (choose x вҲҲ Sub-A) is not available against
  the static-key channel.

The structural reason R1 holds for `c1`: `A^H В· r` performs sedenion
multiplication entry-wise, and the lower projection of any single
sedenion product `(A^H[i][j]) В· r[j]` picks up an UpperГ—Upper
fold-back contribution from `r_upper`. So `r_upper` IS contributing
to the Sub-A channel before `c1` reaches the attacker, satisfying
the spirit of R1. The contribution is computed by the existing
`mat_vec` in `sqt_slwe.py:92` via `s_mul` which uses the full
Cayley-Dickson table вҖ” no code change required.

**Observation (forward-looking, not a current bug):** if a future
API exposes an oracle `oracle(x) вҶ’ ПҖ_A(x В· s)` (e.g. a debug helper,
a constant-time check, or a side-product of an MPC protocol), R1
will bite. Any such helper must additively mix `r_upper`
contribution into the lower projection before return. Recommend
explicit guard in the spec (and a unit test that constructs an
attacker-chosen `x вҲҲ Sub-A`, queries the oracle, and checks that
the residual is non-deterministic / fresh-blinded each call).

### R2 вҖ” r_upper fresh uniform per encryption?

**Verdict: partial violation, two issues.**

#### R2-(a). r is **fresh per call** but flows through Python's `random` (Mersenne Twister), not a cryptographic PRF.

`tools/sqt_slwe.py:232` calls `rand_nonzd()` which uses
`random.randint`. `hybrid_kem/kem_slwe/slwe_toy.py:280` calls
`rand_small_nonzd(...)` which uses `random.choices`. Both routes
ultimately read from the module-level `random` stream.

`slwe_toy._seed_python_random` (lines 160вҖ“164) reseeds Python's
`random` with `int.from_bytes(drbg.generate(16), "big")` вҖ” 128 bits
of DRBG output mapped into MT's seed. The DRBG itself is HKDF-based
(see `hybrid_kem/entropy/drbg.py`), which is fine. But the MT
expansion from that 128-bit seed to the actual `r` samples is **not
cryptographic**: Mersenne Twister state can be recovered from 624
consecutive 32-bit outputs and is fully predictable thereafter.

The addendum's R2 is explicit:

> Derivation of r_upper from a counter or seed is acceptable IF AND
> ONLY IF it is stated explicitly as a PRF assumption in the spec.
> Silent derivation (treating counter-derived as "random") is not
> acceptable вҖ” it is a category error that must be caught at review.

`hybrid_kem/SPEC.md` does not state a PRF assumption over MT. The
current setup is the silent derivation R2 forbids. **Flag.**

#### R2-(b). `tools/sqt_slwe.py:62` calls `random.seed(42)` at module import.

```python
random.seed(42)   # tools/sqt_slwe.py:62
```

Anyone importing `sqt_slwe` directly (without going through the
wrapper's reseed path) inherits a fixed seed across processes вҖ”
deterministic `r`. The wrapper does reseed before every call (good),
but the seed-42 line is a footgun and a hard-coded development
artefact that should not survive into a production build. Two
encryptions back-to-back through the wrapper after a fresh module
import will use the wrapper's reseed, but two encryptions from any
test that imports `sqt_slwe` and calls `encaps` directly will reuse
MT state continuing from seed 42 вҖ” i.e., **r reused across encaps
calls within the same Python process unless the caller reseeds.**

This is a 2-query attack vector per the addendum: "Reuse of r_upper
across two encryptions with the same or related x allows the
attacker to subtract observations and cancel noise."

**Flag.** Both R2-(a) and R2-(b) need spec / implementation
disposition by M. Gifford.

### R3 вҖ” B=1 output blinding before return, no conditional branch?

**Verdict: not implemented. Clearest of the three findings.**

R3 requires:

> Add a uniform random offset `r_blind ~ Uniform({-1, 0, +1}^8)` to
> all Sub-A outputs before returning. B=1 is the minimum effective
> blinding вҖ” eliminates the ZD membership oracle entirely.

No such `r_blind` step exists. Reviewing every encaps return path:

```python
# tools/sqt_slwe.py:239
c1 = [s_add(AHr[i], e1[i]) for i in range(k)]
# tools/sqt_slwe.py:243-245
c2 = (b_dot_r + e2 + m_bit * q2) % p
return c1, c2, m_bit
```

```python
# hybrid_kem/kem_slwe/slwe_toy.py:285-287
c1 = [_slwe.s_add(AHr[i], e1[i]) for i in range(k)]
c2 = (_slwe.norm_inner_k(b_vec, r) + e2 + m_bit * q2) % _slwe.p
return c1, c2, m_bit
```

The error `e1[i]` is a CBD(2) sample over all 16 coordinates and
plays the role of the LWE noise; its lower 8 components are 8
independent CBD(2) draws. The addendum's `r_blind` is **a separate,
additional Uniform({-1,0,+1}^8) offset distinct from the LWE error**
because:

1. Distribution: CBD(2) over {-2,-1,0,1,2} вү  Uniform({-1,0,+1}).
2. Purpose: e1 must stay small for decryption correctness (DFR
   budget); r_blind can be sampled independently of the noise
   budget because it serves a side-channel role.
3. Layering: if e1 is later reduced (e.g. CBD(1) at higher p), the
   blinding role should not silently shrink with it.

The Sub-A projection of `c1[i]` is exposed in pk-byte form by
`_serialize_pk` / `_serialize_ct` (lines 114вҖ“157). The bytes
contain `c1[i][0..7]` directly readable, with no Uniform({-1,0,+1})
mask applied on top. **Flag.**

#### R3 sub-finding: timing branch on near-zero in decaps

`tools/sqt_slwe.py:256` and `hybrid_kem/kem_slwe/slwe_toy.py:293`
(inherits via `_slwe.decaps`):

```python
m_dec = 1 if (q2//2 < v < 3*q2//2) else 0
return m_dec, v
```

This is a near-q/2 branch on a secret-derived value (v contains the
LWE noise residual). In a compiled implementation it would be a
constant-time-decapsulation issue (FO transform / CCA hardening).
In CPython the two arms reduce to the same `LOAD_CONST 0 / 1` and
both flow into the same SHA-256 hash in the wrapper, so the
*Python-level* timing leak is small вҖ” but the algorithmic pattern
is exactly what R3 cautions against and would surface in any
non-Python rewrite. **Flag** as a portability hazard for the
production track.

### Disposition

Per addendum line 105: **do not fix; flag for M. Gifford.** No code
changes have been made by this audit. The fix scope is outside
Brief #02. Recommended next step is a separate brief that:

1. Removes `random.seed(42)` from `sqt_slwe.py` and routes all
   randomness through an explicitly-named PRF whose assumption is
   stated in `SPEC.md` (or replaces MT with `secrets`-backed CSPRNG
   inside the wrapper);
2. Adds the `r_blind ~ Uniform({-1,0,+1}^8)` step to every encaps
   return path, with a unit test that asserts the Sub-A projection
   of `c1` is fresh per call even for identical (pk, m_bit, drbg
   seed) inputs;
3. Rewrites the decaps near-q/2 branch as a constant-time
   integer-comparison style, or documents the Python-interpreter
   relaxation explicitly with an annotation visible to anyone
   porting the code.

The chain-ZD probe itself (`tools/chain_enumerate.py`,
`tools/chain_zd_labelling.py`, the manifest/comparison/report) reads
ZD structure but does **not** generate encaps output and therefore
does not interact with R1/R2/R3. The audit found no side-channel
issue in the chain-ZD code path.
